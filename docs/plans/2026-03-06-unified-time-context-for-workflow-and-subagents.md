# 2026-03-06：三省六部与派生 Subagent 的统一时空上下文需求文档

## 背景

当前三省六部 workflow 中，不同角色与后续派生出来的 subagent，可能并不共享同一套明确、显式、可引用的时间与时区上下文。

这会带来以下问题：

- 不同角色对“现在”的理解可能不一致。
- 对“今天 / 明天 / 近期 / 本周 / 本月 / 刚刚”等相对时间词的解释可能漂移。
- 六部正文与派生 subagent 的取证、论证、引用时间线，可能不在同一个时空平面上。
- 当任务涉及新闻、事件进展、时间线判断、时效性风险、时间窗口分析时，容易出现前后不一致。

因此，需要为：

- 三省六部所有主 agent
- workflow 中所有 step agent
- 由任一主 agent 派生出的 subagent（包括但不限于 research / deep-research / Task 派生子代理）

统一注入一组明确的“时空元信息”，确保全流程工作在同一个时间坐标系内。

---

## 核心目标

本需求的目标不是“让模型更会看时间”，而是：

**把时间与时区从隐式背景，提升为所有 agent / subagent 都能直接读取、直接引用、直接遵守的显式上下文。**

一句话：

> 不管是三省六部主 agent，还是后续派生出来的 subagent，都必须工作在同一个明确、可见、可引用的时空平面上。

---

## 必须统一提供的时空元信息

所有主 agent 与所有派生 subagent，都必须拿到至少以下元信息：

### 1）用户时区

- 用户时区：`Asia/Shanghai`
- 用户所处时区表达：`东八区 / UTC+08:00`

要求：

- 不能只给“本地时间”而不说明时区。
- 不能只给“UTC+8”而不说明这是用户当前工作的默认时区。

### 2）当前绝对时间

必须提供一个明确的、可直接引用的当前时间快照，例如：

- 当前日期
- 当前本地时间（东八区）
- 必要时附 UTC 时间

要求：

- 必须是**绝对时间**，不能只写“现在”。
- 时间表达必须能直接支撑后续 agent 做“今天 / 明天 / 最近 / 截至目前”的判断。

### 3）相对时间解释基准

所有 agent / subagent 在解释以下词语时，都必须以该统一时间快照为准：

- 今天
- 明天
- 昨天
- 近期
- 最近
- 截至目前
- 本周
- 本月
- 当前

要求：

- 不允许各 agent 按各自隐式理解解释。
- 不允许主 agent 和 subagent 对相对时间的起点不同。

---

## 适用范围

本需求适用于以下所有层次：

### A. workflow 顶层角色

- 太子
- 中书省
- 门下省
- 尚书省
- 六部各部

### B. workflow step agent

每一个 step 在执行时获得的 prompt / system prompt / runtime context，都应包含统一时空信息。

### C. 派生 subagent

凡是由主 agent 继续派生出的子代理，都必须继承同一套时空信息，包括但不限于：

- Task 派生子代理
- research 子代理
- deep-research 式取证子代理
- 任何未来新增的 specialized subagent

关键要求：

> subagent 不允许丢失、重置、漂移或重新猜测当前时间与时区。

也就是说：

- 主 agent 看见的“现在”，必须与 subagent 看见的是同一个“现在”。
- 主 agent 使用的默认时区，必须与 subagent 使用的是同一个默认时区。

---

## 功能性要求

### R1. 所有主 agent 必须显式获得统一时空上下文

每个主 agent 在开始工作前，都必须收到统一时空信息。

这组信息必须出现在 agent 可直接读取的位置，而不是依赖隐式环境推断。

### R2. 所有 subagent 必须继承与主 agent 完全一致的时空上下文

任何 subagent 启动时：

- 必须携带同一时区
- 必须携带同一当前时间快照
- 必须沿用同一相对时间解释基准

禁止出现：

- 主 agent 看到的是东八区时间，subagent 却按别的时区理解。
- 主 agent 使用任务启动时的时间，subagent 却重新拿了另一套“当前时间”。
- 主 agent 和 subagent 对“今天”不是同一天。

### R3. 时间信息必须可被正文引用

统一时空信息不能只是“内部存在”，还必须允许 agent 在需要时把它自然地体现在输出中。

例如在涉及：

- 时效性判断
- 时间窗口分析
- 新闻时间线
- 事件是否“已持续多日 / 多周”
- 截至某时点的公开信息

时，agent 应能直接引用这套统一时间上下文。

### R4. 相对时间词默认以统一时空基准解释

对于“今天 / 明天 / 截至目前 / 最近”等词语：

- 若用户没有另行指定，统一按该显式时空信息解释。
- 不允许 agent 自行脑补别的时区或别的当前时间。

### R5. 涉及时效内容时，优先使用绝对时间表述

当任务内容容易因时间理解偏差而出错时：

- 应优先使用绝对日期或绝对时间。
- 必要时把“今天 / 当前 / 目前”转换为明确的东八区日期时间来表达。

目标：

- 减少多 agent 协作下的相对时间歧义。
- 避免 subagent 回填内容与主文稿的时间口径不一致。

---

## 非目标

本需求**不要求**：

- 引入联网校时能力
- 校验系统时钟是否准确
- 自动搜索“最新时间”来源
- 自动比较多个时区版本输出
- 现在就改动所有 prompt 文案细节

本需求只要求一件事：

**统一、显式、稳定地把同一套时区与当前时间元信息提供给所有主 agent 与 subagent。**

---

## 设计原则

### 1）显式优于隐式

不能依赖模型“自己知道现在是什么时间”。

### 2）继承优于重算

subagent 应继承主 agent 的时空上下文，而不是各自重新生成。

### 3）绝对时间优于相对时间

尤其在多 agent 协作和有时效性的任务中，必须尽量降低“今天/最近”的歧义。

### 4）统一优于局部修补

不能只给六部主 agent 加时间信息，而忽略它们派生出来的子代理。

---

## 验收标准（DoD）

以下条件全部满足，才算本需求达成：

### DoD 1

三省六部所有主 agent 都能获得同一套显式时空元信息，至少包含：

- `Asia/Shanghai`
- `UTC+08:00 / 东八区`
- 一个明确的当前绝对时间快照

### DoD 2

任一主 agent 派生出来的 subagent，都能获得与主 agent 完全一致的时空元信息。

### DoD 3

主 agent 与 subagent 对“今天 / 明天 / 截至目前 / 最近”等相对时间词的解释基准一致。

### DoD 4

当任务涉及时效性、时间线、持续时长、窗口期判断时，agent 可以自然输出与东八区当前时间一致的表达，不出现前后矛盾。

### DoD 5

后续若新增其他 specialized subagent，也必须默认纳入同一时空继承机制，而不是额外补丁式接入。

---

## 建议的需求口径（供后续实现时参考）

后续实现时，应把这组元信息视为一等上下文，建议统一命名为类似：

- 当前用户时区
- 当前本地时间（东八区）
- 当前 UTC 时间（可选）
- 相对时间解释基准

重点不是具体字段名，而是以下原则必须成立：

- 主 agent 看得见
- subagent 继承得到
- 输出时可引用
- 整个 workflow 一致

---

## 一句话需求总结

> 为三省六部所有主 agent 与其派生 subagent 统一注入显式的时区与当前时间元信息（至少包括 `Asia/Shanghai`、`东八区/UTC+08:00` 与当前绝对时间快照），确保所有角色工作在同一个时空平面上，并以此统一解释“今天 / 明天 / 最近 / 截至目前”等相对时间表达。
