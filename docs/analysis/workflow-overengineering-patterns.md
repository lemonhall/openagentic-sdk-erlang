# 近期 sessions 的“画蛇添足 / 只讲门禁不干活”模式复盘

目的：基于真实 `workflow_session_id` 的运行历史（`events.jsonl`）归纳可重复的问题模式，并据此调整各部门提示词，让输出更“有产物、有结论、少空话”。

范围：`three-provinces-six-ministries.v1` 近期样本（约近 10~20 个 workflow sessions）。

> 说明：本复盘只讨论“提示词与编排”导致的行为偏差，不讨论 permission gate / 超时 / provider 502 等运行时故障。

---

## 模式 A：把“局势研判固定五件套”硬塞进不相关旨意（最典型：作诗/创作）

现象：
- 在“作诗/对联/文案创作”类旨意里，仍被派发 `claims_ledger / hubu_risk / brief / redlines / scenarios` 等“局势研判”产物。
- 结果是：部门做了很多“合规台账/流程文书”，但对皇上关心的“成品”推进变慢，体验上像“很忙但没干正事”。

根因：
- 尚书省派发提示词里曾把这套产物写成“无条件强制”。

对策（已落地）：
- 在 `shangshu_dispatch` 增加“先判题”，并把五件套限定为仅对“现实世界事件/新闻/局势研判”类旨意生效。
- 创作类旨意明确禁止夹带这套产物。

---

## 模式 B：立案/规划阶段“悬置（待确认/二选一）”过多，导致门下省反复封驳

现象：
- `taizi_intake`、`zhongshu_plan` 经常出现“体裁A/B、口径A/B、待请旨、待确认”等悬置写法。
- 门下省偏好“单一、可执行、可验收”口径，悬置会直接触发 reject，形成循环。

根因：
- 太子立案没有补全默认口径，反而把不确定性扩散成“可选项”。
- 中书省没有把门下省 `required_changes` 做逐条对照吸收，导致“改了但看不出来”。

对策（已落地）：
- `taizi_intake`：强制“默认给口径 + 待确认最多 3 条 + 禁止二选一悬置”。
- `zhongshu_plan`：强调“写到门下省一次准奏”为目标，子任务必须落到 `workspace:deliverables/...`，并追加“门下省意见吸收清单”逐条对照。

---

## 模式 C：部门输出里“原则/门禁/合规措辞”占比过高，正文产物反而缩水

现象：
- 部门输出经常出现大量“红线/闸门/不构成建议/风险提示”，但交付物内容（可直接使用的表格/清单/文稿正文）很短。
- 有时提示词写了“不要输出超长正文”，模型就倾向只写摘要，不落盘正文。

根因：
- 部门提示词对“落盘”虽有要求，但缺少“必须写入文件；主输出只做摘要+路径”的硬约束。
- “不要超长正文”没有配套“正文必须写入 workspace 文件”的指令，导致模型两头都不做。

对策（本轮将落地到六部提示词）：
- 对需要产物的任务：优先 `Write` 到 `workspace:deliverables/...`；若写入被拒，再把正文放到输出里兜底。
- 主输出只保留：处理结果（1 段）、产物路径列表、交接要点；避免重复长篇合规口号。

---

## 模式 D：联网核验的触发逻辑过“保守”，导致“该搜不搜”

现象：
- 部门普遍把 WebSearch/WebFetch 当作“事实核查”，而不是“资料/背景/来源支撑”。
- 当皇上未明确说“请核实”，部门就完全不搜，导致内容缺少事实支撑，显得空泛。

根因：
- 提示词把联网搜索与“质疑旨意”绑定在一起。

对策（建议，需与你确认是否启用）：
- 将“WebSearch/WebFetch”分成两类：
  1) **核验**（会影响真假判断）：仍按“皇上明确求证/传闻”才触发；
  2) **背景资料/常识支撑**（不挑战旨意前提）：允许在需要“对外简报/引用来源”时触发，最低限度补 2~3 条来源用于支撑论述。

---

## 结论

三省六部制的优势在“可拆解、可落盘、可复盘”；但提示词若把“合规/门禁/流程文书”写成主角，就会出现“忙而无功”的体验。

本轮优先修复的方向是：
- 把“成品落盘”写成硬目标；
- 把“门禁/合规”压到最小必要且可核对；
- 用“判题”避免不相关产物污染。
