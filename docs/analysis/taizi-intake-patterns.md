# 太子立案常见问题与改进要点（面向 `taizi_intake`）

目标：减少“模糊/悬置/画蛇添足”的立案写法，降低后续中书省反复改稿、门下省封驳概率；**不通过更改门下省提示词**来达成。

分析时间：2026-03-06（近期 workflow sessions 样本）

---

## 近期样本中出现的典型问题（高频）

### 1) 过多“待确认/二选一”，导致后续无法落笔

常见写法：

- “体裁与格律：五言/七言；绝句/律诗/古风：待确认”
- “押韵口径：押何韵、是否一韵到底：待确认”
- “具体篇幅标准/对仗数量：待确认”

后果：

- 中书省容易照抄“待确认”，继续写成“二选一/请示”，门下省经常封驳“不够确定/不可验收”。

### 2) 把流程与验收写成主角，交付物反而不清晰

立案里如果强调“校对/闸门/流程”过多，而没有在开头锁定：

- 唯一交付物是什么
- 写到 `workspace:deliverables/...` 的哪个文件
- 皇上打开后能直接读到什么（结构/顺序）

就会在后续汇总时变成“执行过程汇总”，而不是“结论/成品”。

### 3) 对创作类任务的“加码”倾向

例如把“对仗/押韵/格律/模板/反向限制”写得过重，会显著提高六部并行产出难度，并诱发门下省以“难以执行/不宜加码”封驳。

---

## 改进要点（太子立案的写法策略）

### A. 缺省要补全：默认给一个可执行口径

只要不涉及安全/合规风险，就应该在立案中明确：

- 默认体裁/篇幅/格式（单一口径）
- 默认交付物路径（`workspace:deliverables/...`）
- 默认排版（打开即可读/可宣读）

### B. “待确认”必须稀缺且真实关键

建议硬规则：

- 全篇最多 3 条 “待确认/缺口”
- 且每条必须解释：缺了会导致“无法开写/无法验收/容易误伤合规”的原因

### C. 把“门下省偏好”前置为写作约束

门下省对规划文书的稳定偏好是：

- 反对悬置（不接受“二选一/可选/待请旨”）
- 偏好可核对（行数/字数/结构/路径/格式）而非主观评价

太子立案中应把“单一口径 + 可验收”的要求写成对中书省的交接约束。
