# 门下省常见封驳模式与改进策略（面向中书省）

目的：提高 `menxia_review` 对 `zhongshu_plan` 的一次准奏率，**不更改门下省提示词**，仅通过改进中书省文书写法与可验收性来提升通过率。

分析时间：2026-03-06（以 `E:\openagentic-sdk\sessions\*\events.jsonl` 近期样本为依据）

---

## 观察到的高频封驳点（聚类）

### 1) “悬置/多选项/待请旨”导致不可执行

门下省经常对以下写法封驳：

- “体裁 A/B 二选一”“押韵口径 A/B 二选一”“是否统一同韵未定”
- “可选/建议/视情况”“若上谕补充则切换”

门下省倾向于：**要么给出单一口径，要么明确默认口径并立即执行**；除非皇上原旨明确要求备选，否则不接受悬置。

### 2) “加码”与“过度规制”引发反感

在“作诗/文案/呈递稿”类任务中，中书省若提出过强的格律/对仗/反向限制，容易被认为：

- 让六部难以按时交付
- 验收主观或不可核对
- 把“制度与清单”压过“呈送成品”

门下省倾向于：**以可读/可宣读的成品优先**，规制只保留最低限度的格式统一与明显错误纠正规则。

### 3) 交付物不够“皇上可直接阅读/宣读”

高频要求包括：

- 唯一交付物路径必须锁死在 `workspace:deliverables/...`
- 最终稿必须“打开即读、可直接朗读/宣读”，而不是“流程汇总 + 验收清单”

### 4) DoD/验收缺“可逐条核对”或表述过主观

门下省喜欢：

- 行数/字数/换行规则/标题模板/是否标点等可机械核对项

门下省不喜欢：

- “不宜朗读”“古雅”“稳妥”等无法客观核对的验收条件

## 面向中书省的写作改进策略（可直接落到 `zhongshu_plan`）

### A. 先锁死“唯一交付物”与路径

在 `总体方案` 第一段即写明：

- 唯一交付物：是什么（例：六部诗稿总卷/宣读稿/对外回信）
- 保存路径：`workspace:deliverables/<file>.md`
- 内容结构：打开后从上到下是什么（例：序言→六部成品→附注）

### B. 默认单一口径，不给二选一

除非皇上明确要求提供备选，否则中书省应：

- 直接裁定体裁/篇幅/押韵口径（如需押韵，明确押韵句位与“统一/各自”策略）
- 将其写入 DoD（可核对）

如确有不确定，只能写：

> “默认按 A 执行；如陛下另谕则改”

但仍必须给出 **A 的完整可执行规格**。

### C. 子任务拆解必须产出“可并入总稿的成品段落”

对六部的子任务，不要只写“做一个报告/写一首诗”，而要写：

- 输出形态：成品段落（标题+正文），可直接拼进总稿
- 输出路径：固定在 workspace 内某文件/某 section
- 自检最小项：行数/字数/格式

### D. 增加“门下省意见吸收清单”（逐条对照）

当输入包含门下省 `reject` 的 `required_changes` 时：

- 在 `风险与假设` 末尾追加 `门下省意见吸收清单`
- 每条 required_change 用一句话回应“已如何落实/落在文档哪一节”

该段的价值在于：让门下省能快速扫描并准奏（减少反复封驳）。

## 备注：为什么不把“主会话历史”全塞给每一步？

workflow 设计是“Controller 全局注入 + Input 显式绑定”，用 DSL 控制各 step 可见范围，以提高可控性与可验收性。详见：`docs/design/workflow-context.md`。
